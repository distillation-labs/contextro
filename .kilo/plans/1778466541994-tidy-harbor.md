# Contextro Multi-Agent Packaging Plan

## Research Question
How do we package Contextro once and deliver it cleanly to Claude Code, Codex/OpenAI, GitHub Copilot, Cursor, Kiro, and similar agents without duplicating the workflow content or drifting across formats?

## Current Baseline
- `packages/skills` is already the canonical skill source. It ships `dev-contextro-mcp` and `contextro-quickstart` from `packages/skills/skills/*/SKILL.md`.
- The current installer writes to `.agent/skills`, `.github/skills`, `.kiro/skills`, and `.opencode/skills`.
- The repo already has root-level `CLAUDE.md` and `AGENTS.md` guidance.
- The benchmark/experiment flow already exists in `packages/skills/bin/experiment.mjs` and the CLI tests cover install/uninstall/list/info/benchmark.

## Facts
| Surface | Native support from research | What it means for us |
|---|---|---|
| Claude Code | Plugins, project skills in `.claude/skills`, plugin-root `skills/`, `CLAUDE.md`, hooks, MCP | Claude should get a true plugin path plus a repo-local skill path. `.agent/skills` is not the current official path. |
| GitHub Copilot | `.github/copilot-instructions.md`, `.github/instructions/*.instructions.md`, `.github/prompts/*.prompt.md`, `.github/skills/`, MCP, CLI plugins | Copilot needs repo instructions first, plus skills and optional plugin packaging where it helps. |
| Codex/OpenAI | `AGENTS.md` as a table of contents, structured `docs/` as the system of record | Codex should be served with repo docs and concise instructions, not a giant monolith. |
| Cursor / Kiro / OpenCode | Native instruction/skill/rule surfaces exist, but exact file conventions must be verified before implementation | Treat these as adapter targets, not as the same packaging shape as Claude or Copilot. |

## Inferences
- There is no single universal plugin format across these tools. The lowest-risk architecture is one canonical content source plus per-platform renderers.
- The current package is useful, but the Claude output path needs normalization to match current Claude Code docs.
- Codex does not need a plugin in the Claude/Copilot sense. It needs repo legibility: short AGENTS guidance, linked docs, and MCP setup that is easy to discover.
- Cursor and Kiro should be handled as first-class adapter targets, but only after their current on-disk conventions are verified.

## Adopt / Adapt / Avoid
| Strategy | Verdict | Why |
|---|---|---|
| Canonical skill manifest + generators | Adopt | Keeps content stable and makes platform-specific output deterministic. |
| Claude and Copilot true plugin packaging | Adapt | Use when the host has an actual plugin model. |
| Repo-local instructions and docs for Codex/Cursor/Kiro-style surfaces | Adopt | These platforms are better served by native instructions/rules than by fake plugins. |
| One giant cross-vendor plugin artifact | Avoid | Too brittle, too many naming and discovery differences. |
| Hand-written copies of the same guidance per platform | Avoid | Guarantees drift and increases maintenance cost. |

## Proposed Plan
### Phase 0: lock the platform matrix
- Verify the exact on-disk conventions for Cursor and Kiro before writing adapters.
- Decide which surfaces are first-class in v1: Claude plugin, Claude project skills, Copilot repo instructions/skills, Codex repo docs, and the existing Kiro/OpenCode outputs.
- Decide whether `.agent/skills` stays only as a compatibility alias or is removed from the primary docs.

### Phase 1: define one source of truth
- Add a small manifest schema for each skill: name, description, trigger text, required tools, scope, and output targets.
- Keep the actual skill prose in one place and generate everything else from that.
- Mark skills as either always-on guidance or manual-entry workflows.

### Phase 2: build renderers
- Claude renderer: emit `.claude/skills/<skill>/SKILL.md` and a plugin-ready tree under `.claude-plugin/` when plugin packaging is requested.
- Copilot renderer: emit `.github/copilot-instructions.md`, `.github/skills/<skill>/SKILL.md`, and prompt files where a reusable prompt is useful.
- Codex renderer: emit/refresh `AGENTS.md` plus linked docs, keeping the file short and navigational.
- Kiro/Cursor/OpenCode renderer: emit the closest native instruction or skill layout after the convention check in Phase 0.

### Phase 3: validate the adapters
- Add snapshot tests for each renderer so generated files stay stable.
- Extend CLI tests to cover every supported platform directory, including the Claude-native path.
- Add one temp-repo smoke test that installs or emits all targets and verifies discovery by path, not just file creation.

### Phase 4: publish and migrate
- Update `README.md`, root docs, and changelog with the platform matrix and install instructions.
- Add migration notes from the current `.agent/skills` path to the Claude-native path.
- Call out which surfaces are true plugins and which are repo-local instruction bundles.

## Top Experiments
1. Adapter matrix smoke test
- Generate each target surface into a temp repo.
- Success criterion: all expected files are created in the right paths, with no manual edits.

2. Claude path normalization
- Switch the Claude output target to `.claude/skills` and, if needed, add plugin packaging separately.
- Success criterion: the generated skill is discoverable by Claude Code conventions and existing tests pass.

3. Copilot instruction coverage
- Emit `.github/copilot-instructions.md` plus one reusable prompt or skill file.
- Success criterion: the repo contains a minimal Copilot-native entrypoint and the docs point to it clearly.

4. Codex legibility check
- Keep `AGENTS.md` short and use docs links for deeper detail.
- Success criterion: AGENTS stays concise and the repo can still be understood from the top-level map.

## Recommendation
Ship a manifest-driven adapter system, not a separate hand-authored package per vendor.
- Use true plugins only where the host supports them cleanly, especially Claude Code and Copilot CLI.
- Use repo-local instructions/rules/docs for Codex, Cursor, Kiro, and any surface without a stable plugin model.
- Fix the Claude path mismatch early, then expand outward from the shared manifest and validation harness.
