Contextro: Token-Efficient Repository Discovery
for MCP-Based Coding Workflows

Japneet Kalkat
Distillation Labs
japneet@distillation.ai

Abstract

Repository discovery is a major source of token overhead in MCP-based coding workflows. This report evaluates Contextro, a proprietary local code-intelligence server for repository search, symbol navigation, structural analysis, and persistent memory, using a 60-task paired study on an 8,496-file production TypeScript monorepo. Relative to a bounded grep-based pipeline with repository-wide grep plus up to five file reads, the Contextro pipeline reduced total token consumption from 328,272 to 14,407, reduced median latency from 448.35 ms to 5.21 ms, and eliminated file reads. On the directly comparable 39-task subset, token consumption fell to 6,638, a 98.0% reduction; across 100 deterministic bootstrap subsets, reduction remained between 97.24% and 98.48%. Supplemental public verification now includes reruns on the open src/ tree, a 39-task stronger-baseline rerun on a 1,179-file public proxy repository, and an 8-task deterministic edit-correctness benchmark. These results bound discovery efficiency and scripted edit correctness rather than full autonomous coding quality.

Keywords: Model Context Protocol, repository discovery, code intelligence, semantic retrieval, structural search, technical evaluation

1. Introduction

Repository discovery remains a costly phase in MCP-based coding workflows. Before a client can act on the correct symbol, file, or interface, it typically executes multiple broad searches, opens several files, and accumulates context that is only partially relevant to the task. In the paired study reported here, a representative grep-based workflow consumed 5,524 tokens and took 1.11 s, whereas the corresponding indexed find_symbol call returned in 5.66 ms with 65 tokens.

Contextro addresses this inefficiency by replacing file-by-file exploration with structured repository access. It exposes repository search, symbol navigation, relationship tracing, impact analysis, structural search, history lookup, and persistent memory through a local MCP server. The operating premise is straightforward: when discovery tasks can be answered directly, the client does not need to spend context budget reconstructing repository structure from raw files.

This report makes four bounded claims. First, a local code-intelligence layer can materially reduce the token cost of repository discovery. Second, the resulting interface can deliver low latency on a production-scale repository for common lookup tasks. Third, the effect remains visible under stricter analysis that separates directly comparable discovery tasks from MCP-native capabilities with no lexical analogue. Fourth, public supplementary artifacts can expose stronger local-tool baselines, exact MCP tool schemas, and deterministic edit-correctness checks without redistributing the private study corpus.

2. Related Work

This work sits within a broader literature on contextual retrieval, hybrid search, and prompt-budget management. Prior work from Anthropic, NVIDIA, Liu et al., and OpenAI argues that retrieval systems perform best when they combine multiple signals, rank relevant evidence near the top of the context window, and avoid transmitting unnecessary text. The present report applies that general logic to repository discovery, where the cost of locating the right code often dominates the cost of reasoning over it.

3. System Design

Contextro operates as a local MCP service positioned between the coding client and the repository. Rather than requiring exploratory file reads, it returns structured answers to discovery questions, including identifiers, locations, relationships, and compact supporting excerpts. The objective is to shift discovery from file traversal toward targeted retrieval.

3.2. Retrieval and Response Design

The retrieval layer combines semantic, lexical, and structural signals. Semantic retrieval supports concept-level matching, lexical retrieval supports exact identifiers and strings, and graph-based traversal supports questions about callers, callees, and impact. Returned material is filtered and abbreviated so that the interface favors direct answers over large context transfers.

The implementation is local and single-process. It integrates vector indexing, lexical retrieval, structural analysis, syntax-aware search, history lookup, and memory in one service boundary. That integration matters operationally because it eliminates the need to orchestrate several external systems during interactive discovery.

4. Methodology

The evaluation uses a paired two-condition design on a single production repository. Each task is executed once with a bounded grep-based discovery workflow and once with the Contextro pipeline. The full task inventory contains 60 tasks across 16 categories. Among these, 39 tasks have direct file-centric analogues and therefore support strict paired comparison; the remaining 21 tasks represent MCP-native capabilities with no meaningful file-centric baseline.

4.1. Study Design

The primary comparison pairs a bounded grep-based pipeline with the Contextro pipeline. For each directly comparable task, the grep-based pipeline records repository-wide grep output and the contents of up to five matching files in the token estimate. The Contextro pipeline executes one MCP tool call against a pre-indexed repository state. Tasks without a meaningful file-centric analogue are retained in the full inventory as MCP-native capabilities, but they are excluded from the directly comparable subset and from the bootstrap robustness analysis. Both study arms are scripted pipelines rather than autonomous LLM agents.

This design isolates discovery cost rather than downstream coding quality. It compares a file-centric grep-based pipeline with a structured retrieval interface under a fixed task inventory, shared repository, and common token accounting rule.

Exact JSON request/response schemas and example payloads for 30 tools are exported as a machine-readable artifact.

5. Results

5.1. Headline Outcomes

Across the full 60-task inventory, total token consumption fell from 328,272 in the bounded grep-based pipeline to 14,407 in the Contextro pipeline. Mean tokens per task fell from 5,471.2 to 240.1, median latency fell from 448.35 ms to 5.21 ms, and mean files read fell from 2.5 per task to 0. The primary control is a bounded grep-based pipeline, not an LSP-backed or autonomous coding baseline. The reported gains therefore characterize discovery efficiency rather than universal coding quality.

The stricter comparison is the 39-task subset with direct file-centric analogues. On that subset, token use fell to 6,638, a 98.0% reduction, while median latency fell from 498.89 ms to 4.89 ms and mean tool calls fell from 4.9 to 1.0. The gap between the 60-task and 39-task summaries reflects the presence of 21 MCP-native tasks that expose analysis surfaces absent from the grep-based pipeline.

The proxy rerun is intentionally harder than the original open-src/ checks. It uses a larger redistributable repository and a stronger local-tools pipeline composed of git grep, regex search, direct file reads, and git log. Contextro still preserved 39/39 task success while reducing mean tokens by 69.1%, reducing mean latency from 36.82 ms to 8.89 ms, and reducing mean tool calls from 2.95 to 1.0.

The deterministic edit benchmark closes part of the end-to-end gap by requiring correct target discovery, exact rewrite application, and post-edit syntax validity. Both pipelines achieved 8/8 successful edits. On these narrowly scripted constant-update tasks, Contextro was faster but returned larger structured payloads, so the result should be read as correctness and orchestration evidence rather than as a new token-efficiency headline.

6. Discussion

The most important interpretive point is that the result becomes stronger, not weaker, under the stricter directly comparable subset. The full 60-task inventory shows a 95.6% token reduction, while the 39-task subset shows a 98.0% reduction because it excludes MCP-native capabilities that have no direct file-centric baseline. This separation clarifies the distinction between replacing file-centric discovery and adding entirely new analysis surfaces.

The bootstrap robustness analysis strengthens that interpretation. Across 100 deterministic task subsets, reduction never fell below 97.24% and the median remained 97.87%. These values do not establish universal generalization across repositories, but they do indicate that the result is not driven by a small number of unusually favorable tasks.

Latency should be interpreted similarly. Common lookup operations are effectively instantaneous, while a few repository-wide analyses remain expensive. The practical result is not uniform sub-10 ms performance; it is that routine discovery operations remain fast while deeper structural queries continue to return structured answers at substantially lower token cost than exploratory file reads.

The supplemental public artifacts narrow two obvious credibility gaps. First, the 1,179-file proxy repository replaces the tiny open-tree rerun with a redistributable codebase large enough to exercise symbol search, graph traversal, regex search, and seeded git-history tasks under a stronger local-tools pipeline. Second, the deterministic edit benchmark shows that the retrieved target can be carried through exact code changes and syntax validation. Neither supplement transforms the paper into a universal agent benchmark, but both make the publication claims easier to audit externally.

7. Limitations

The findings in this report are bounded. They describe repository discovery efficiency under a paired task protocol, not general-purpose coding quality under all workflows or repositories.

9. Conclusion

Contextro demonstrates that a local code-intelligence layer can reduce the token and latency cost of repository discovery in MCP-based coding workflows. In the paired evaluation reported here, the full 60-task inventory reduced total tokens by 95.6%, while the stricter 39-task directly comparable subset reduced them by 98.0% and remained above 97.24% across 100 bootstrap task subsets. Supplemental public artifacts show that the direction of the effect survives a stronger local-tools proxy rerun and that the resulting discovery outputs can drive correct scripted code changes, although the paper still does not claim superiority on full autonomous coding benchmarks. The primary conclusion remains narrow but durable: structured repository access can materially outperform bounded file-centric discovery for repository search and navigation.

Appendix B. MCP Tool API Contracts

This appendix summarizes the externally visible MCP tool contracts used by the evaluation. Parameter lists and response fields are described at the interface level. All tools return an MCP ToolResult whose structured_content field carries the JSON payload summarized below; errors are returned as {"error": "…message…"}. The exact machine-readable companion file contextro-tool-api-schemas.json publishes JSON Schema-compatible input and output definitions, annotations, and example request/response payloads for 30 tools.

B.1. Discovery and Navigation

search(...) returns ranked retrieval results with file locations, symbol labels, and compact code excerpts.

find_symbol(...) returns exact or fuzzy symbol matches, including definition metadata and local relationship counts.

find_callers(...) and find_callees(...) return direct relationship sets for the requested symbol.

explain(...) returns symbol metadata together with explanatory prose and supporting snippets.

impact(...) returns a transitive impact set summarizing affected symbols, files, and directories.

B.2. Project Intelligence

overview(), architecture(), focus(...), and restore() return project- or file-level summaries intended to orient the client within the repository.

analyze(...), dead_code(), circular_dependencies(), test_coverage_map(), and audit() return higher-level analysis summaries rather than direct retrieval results.

B.3. Git and History

commit_search(...) returns ranked commit matches for a query, while commit_history(...) returns chronological commit lists. repo_status() summarizes the multi-repository registry state.

B.4. Code Intelligence

code(operation, ...) dispatches syntax-aware operations such as symbol lookup, pattern search, and codebase overview generation. Output structure varies by operation but consistently returns result or match collections.

B.5. Memory and Session

remember(...), recall(...), and forget(...) manage persistent memory entries. knowledge(...), session_snapshot(), and compact(...) manage broader session and knowledge state.

B.6. Server Utilities

status(), health(), and index(...) expose operational server state. retrieve(...) fetches sandboxed large responses by reference, and introspect(...) returns runtime tool schemas and settings.
